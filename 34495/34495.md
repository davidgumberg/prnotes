# [#34495](https://github.com/bitcoin/bitcoin/pull/34495) Replace boost signals with minimal compatible implementation
_All code comments in `[]` are my own._

## Background

Bitcoin Core currently depends on a small subset of boost::signals2 primarily
used for callbacks.

## Problem

Bitcoin Core would like to minimize it's reliance on external dependencies,
especially large and not very actively maintained ones like boost::signals2, and
because the set of behaviors Bitcoin Core depends on such a small subset of the
signals2 API, we can have our own simple signals class that is easy to reason
about, test, and review.

## Solution

### [signals: Use a lambda to avoid connecting a signal to another signal](https://github.com/bitcoin/bitcoin/pull/34495/commits/fd5e9d990431a6af08dd99b25d1e17c6c9818b4d)

	This is undocumented and unspecified Boost behavior that happens to work as
	intended for now, but could break at any point in the future.

Currently Bitcoin Core depends on undocumented signals2 behavior that would also
probably increase the complexity of our own implementation: Connecting a signal
to another signal.

The context for this is the `NotifyCanGetAddressesChanged` signal, which is used
to bubble to the GUI whenever whether or not a wallet can generate receiving
addresses changes, this can change in four circumstances:

Whenever a ScriptPubKeyManager[^1] is [activated][spkm-activated] (e.g.
`importdescriptors` RPC) or [deactivated][spkm-deactivated] in a wallet, whether
or not the wallet [`CanGetAddresses`][cangetaddresses]. This also is signalled
whenever an spkm tops up from the db as both can change whether or not there are
available addresses left to give out, this signal is exclusively used by the GUI
to decide whether or not to enable the `Receive` button for a wallet which is
used to generate and display an address to the user, this was introduced in
[#15225][pr-15225].

Since this can change both at the level of the wallet, e.g. adding or removing
an spkm can change whether or not the wallet is able to provide addresses, and
at the level of the SPKM after `TopUp()` since:

```cpp
bool DescriptorScriptPubKeyMan::CanGetAddresses(bool internal) const
{
    // We can only give out addresses from descriptors that are single type (not combo), ranged,
    // and either have cached keys or can generate more keys (ignoring encryption)
    LOCK(cs_desc_man);
    return m_wallet_descriptor.descriptor->IsSingleType() &&
           m_wallet_descriptor.descriptor->IsRange() &&
           (HavePrivateKeys() || m_wallet_descriptor.next_index < m_wallet_descriptor.range_end);
}
```

Basically, `CanGetAddresses()` can change in the lifetime of an SPKM if the
`next_index` moves (after address acquisition), or if the end_range moves (after
topup). This is an SPKM level detail so it has to be signaled there and bubble
up to the wallet before bubbling to the GUI.

[^1]: Coarse description: ScriptPubKeyManagers are the objects that represent
    springs or sources of "addresses"... really `scriptpubkey`s, and they might
    know something about how to sign for things or who to go talk to
    (SigningProvider) to get signatures for `scriptpubkey`'s .

The current code does this by directly connecting the SPKM signal to the wallet
signal, i.e. when the spkm signal fires, the wallet signal fires, ur  not sposed
to do that, so just make a lambda that fires the signal instead!! !lol

For more context, please see [#26422][issue-26422]

### [signals: remove forward-declare for signals](https://github.com/bitcoin/bitcoin/pull/34495/commits/9ade3929aaa9b43577737f37404e28fd858b808d)

	This eases the transition to a replacement signals implementation

My understanding of this topic is very limited (deliberately, I hope?) but in
[#13634][pr-13634], inclusion of `<boost/signals2/*.hpp>` was removed from all
header files and moved to a single implementation file, since inclusion in
header files means that the templates were reprocessed when compiling any file
that included the header which included `<boost/signals2/*.hpp>`, which resulted
in slowing down compilation. this necessitated forward-declaring the
`boost::signals2::connection` type.

```diff
-#include <boost/signals2/last_value.hpp>
-#include <boost/signals2/signal.hpp>
-
 class CWallet;
 class CBlockIndex;
+namespace boost {
+namespace signals2 {
+class connection;
+}
+} // namespace boost
```


As a side note, since then, a number of header files have come to include boost
signals2 headers and I believe this negates partially the performance
improvement of #13634, e.g. in [src/wallet.cpp][walletsignalsincl].

This can be dropped in favor of just including `<btcsignals.h>` in these header
files, since the compilation performance drop won't matter for this transient
period where `<btcsignals.h>` inclusion is boost signals2 inclusion.

##### return value order

By default, and in the current implementation exclusively, the `combiner` used
by boost:signals2 is optional last value. A combiner decides what to return on
signal firing given that there can be 0, 1, or >1 callbacks connected to a
signal, optional last value, returns an optional that is `std::nullopt` if
nothing is called, and returns the value of the last called callback if
something is returned.

```cpp
/* The result of the signal invocation should always be the result of the last
 * enabled callback.
 */
BOOST_AUTO_TEST_CASE(return_value)
{
    btcsignals::signal<bool()> sig0;
    // [ shorthand for btcsignals::signal<bool()>::result_type ret; ]
    decltype(sig0)::result_type ret;
    ret = sig0();
    // [ std::nullopt since nothing was connected. ]
    BOOST_CHECK(!ret);
    {
        // [ It returns for 1! ]
        btcsignals::scoped_connection conn0 = sig0.connect(ReturnTrue);
        ret = sig0();
        BOOST_CHECK(ret && *ret == true);
    }
    // [ std::nullopt again since nothing was connected when we fired. ]
    ret = sig0();
    BOOST_CHECK(!ret);
    {
        // [ connect up some stuff, true first, false second... ]
        btcsignals::scoped_connection conn1 = sig0.connect(ReturnTrue);
        btcsignals::scoped_connection conn0 = sig0.connect(ReturnFalse);
        // [ false connected last so that's the return... ]
        ret = sig0();
        BOOST_CHECK(ret && *ret == false);
        // [ Disconnect false and expect true since that's still connected... ]
        conn0.disconnect();
        ret = sig0();
        BOOST_CHECK(ret && *ret == true);
    }
    ret = sig0();
    BOOST_CHECK(!ret);
}
```

##### Thread safety

```cpp
/* Test the thread-safety of connect/disconnect/empty/connected/callbacks.
 * Connect sig0 to an incrementor function and loop in a thread.
 * Meanwhile, in another thread, inject and call new increment callbacks.
 * Both threads are constantly calling empty/connected.
 * Though the end-result is undefined due to a non-deterministic number of
 * total callbacks executed, this should all be completely threadsafe.
 * Sanitizers should pick up any buggy data race behavior (if present).
 */
BOOST_AUTO_TEST_CASE(thread_safety)
{
    btcsignals::signal<void()> sig0;
    std::atomic<uint32_t> val{0};
    // [ Connect one incrementer to the signal... ]
    auto conn0 = sig0.connect([&val] {
        val++;
    });

    // [ Construct a thread that... ]
    std::thread incrementor([&conn0, &sig0] {
        // [ Fires sig0 1000x ]
        for (int i = 0; i < 1000; i++) {
            sig0();
        }
        // Because these calls are purposely happening on both threads at the
        // same time, these must be asserts rather than BOOST_CHECKs to prevent
        // a race inside of BOOST_CHECK itself (writing to the log).
        assert(!sig0.empty());
        assert(conn0.connected());
    });

    // [ Construct a thread that... ]
    std::thread extra_increment_injector([&conn0, &sig0, &val] {
        static constexpr size_t num_extra_conns{1000};
        std::vector<btcsignals::scoped_connection> extra_conns;
        extra_conns.reserve(num_extra_conns);
        // [ 1000x ... ]
        for (size_t i = 0; i < num_extra_conns; i++) {
            BOOST_CHECK(!sig0.empty());
            BOOST_CHECK(conn0.connected());
            // [ Connects another increment callback... ]
            extra_conns.emplace_back(sig0.connect([&val] {
                val++;
            }));
            // [ Fire signal ... ]
            sig0();
        }
    });
    // [Block until incrementor completes...]
    incrementor.join();
    // [Block until extra_incrementor_injector completes...]
    extra_increment_injector.join();
    conn0.disconnect();
    BOOST_CHECK(sig0.empty());

    // sig will have been called 2000 times, and at least 1000 of those will
    // have been executing multiple incrementing callbacks. So while val is
    // probably MUCH bigger, it's guaranteed to be at least 3000.
    // [ Is this not actually 1000 + {2 + 3 + ... + 999 + 1000 + 1001} = 502,500?  ]
    //
    BOOST_CHECK_GE(val.load(), 3000);
}
```

```cpp
    bool recursive_callback_ran{false};

        BOOST_CHECK(!sig0.empty());
        // [ That sets the non-recursive flag (nrf) to true... ]
        nonrecursive_callback_ran = true;
    });
    // [ nrf is false... ] 
    BOOST_CHECK(!nonrecursive_callback_ran);
    // [ fire ... ] 
    sig0();
    // [ nrf is true!  ]
    BOOST_CHECK(nonrecursive_callback_ran);
    BOOST_CHECK(conn0.connected());

    // [ manually reset nrf ]
    nonrecursive_callback_ran = false;
    // [ add another signal that sets nrf true and then dc's itself.. ]
    conn1 = sig0.connect([&] {
        nonrecursive_callback_ran = true;
        conn1.disconnect();
    });
    // [ state table, both conns setup... nrf false ... ]
    BOOST_CHECK(!nonrecursive_callback_ran);
    BOOST_CHECK(conn0.connected());
    BOOST_CHECK(conn1.connected());
    // [ fire! ]
    sig0();
    // [ nrf is true... conn0 alive, conn1 autodestructed.. ]
    BOOST_CHECK(nonrecursive_callback_ran);
    BOOST_CHECK(conn0.connected());
    BOOST_CHECK(!conn1.connected());

    // [ reset nrf = false... ] 
    nonrecursive_callback_ran = false;
    // [ new conn1, it connects a signal conn2... ]
    conn1 = sig0.connect([&] {
        // [ conn2 dc's conn0 and dc's itself and sets recursive callback (rf =
        //   true... ]
        conn2 = sig0.connect([&] {
            BOOST_CHECK(conn0.connected());
            recursive_callback_ran = true;
            conn0.disconnect();
            conn2.disconnect();
        });
        nonrecursive_callback_ran = true;
        conn1.disconnect();
    });
    // [ everything false! ] 
    BOOST_CHECK(!nonrecursive_callback_ran);
    BOOST_CHECK(!recursive_callback_ran);
    // [ conn0, conn1, !conn2 ]
    BOOST_CHECK(conn0.connected());
    BOOST_CHECK(conn1.connected());
    BOOST_CHECK(!conn2.connected());
    sig0();
    // [ nrf should be true ]
    BOOST_CHECK(nonrecursive_callback_ran);
    // [ rf is false since conn1 connects conn2, and conn2 sets rf=true, but
    //   conn2 is not connected yet. ]
    BOOST_CHECK(!recursive_callback_ran);
    BOOST_CHECK(conn0.connected());
    // [ conn1 dc's itself and connects conn2. ]
    BOOST_CHECK(!conn1.connected());
    BOOST_CHECK(conn2.connected());
    sig0();
    // [ conn2 runs! sets rf = true ]
    BOOST_CHECK(recursive_callback_ran);
    // [ everything dc since conn2 dc 0 and 2 ]
    BOOST_CHECK(!conn0.connected());
    BOOST_CHECK(!conn1.connected());
```
